# 项目用途
本项目用于软件工程本科毕业设计论文的审查、修改建议生成和正式化改写。

# 工作范围
当处理论文正文、章节草稿、评阅意见、修改稿或相关审查任务时，优先参考 `thesis-skills/` 目录下的本地 skill 定义。

# 本地 skill 使用顺序
默认按以下顺序组织任务：

1. `se-thesis-review-orchestrator`
   - 识别当前内容属于哪一类章节或段落
   - 判断该内容在毕业论文中的功能定位
   - 决定本次审查应重点关注结构、逻辑、专业性还是语言表达
   - 组织最终输出结构

2. `software-engineering-thesis-checker`
   - 从软件工程专业角度审查需求分析、系统设计、系统实现、系统测试等内容
   - 识别“功能介绍化”“项目说明书化”“操作手册化”“测试截图化”等问题
   - 判断是否缺少角色分析、业务流程、模块设计、数据库设计、接口设计、测试用例与结果分析等关键内容

3. `academic-style-rewriter-zh`
   - 将原文改写为正式、规范、自然的本科毕业论文表达
   - 去除口语化、宣传化、模板化和明显 AI 痕迹
   - 保持原意，不虚构功能、数据、实验结果或技术细节

4. `thesis-review-output-formatter`
   - 将审查结果整理为固定结构：
     1. 总体评价
     2. 存在的问题
     3. 修改建议
     4. 修改后的正式版本
     5. 补充建议

# 全局要求
- 不得把项目介绍当作毕业论文写作
- 不得把功能说明当作需求分析
- 不得把页面说明当作系统设计
- 不得把操作步骤当作系统实现
- 不得把测试截图当作测试分析
- 不得虚构系统功能、数据库结构、接口设计、测试数据、实验结论或创新点
- 必须结合章节功能定位进行判断
- 输出必须使用正式、规范、清晰的中文
- 对不符合毕业论文规范的问题要直接指出，不得弱化

# 默认输出格式
除非用户明确要求其他格式，否则按以下结构输出：
1. 总体评价
2. 存在的问题
3. 修改建议
4. 修改后的正式版本
5. 补充建议（如需要）

# 使用提示
当用户提供某一具体章节时：
- 先由总控 skill 判断章节类型
- 再调用专业性审查与语言改写规则
- 最后统一整理输出

当用户只要求润色时：
- 仍需保留必要的问题提示
- 不仅修改语言，还要检查是否存在明显结构或专业性问题
