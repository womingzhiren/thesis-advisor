# AGENTS.md

## 项目定位

本项目用于构建一组面向软件工程本科毕业论文修改指导的 Codex / ChatGPT Skills。

这些 Skills 的目标不是帮助学生完成选题、开题报告、任务书或从零编写毕业论文，而是辅助指导老师对已经形成的本科毕业论文或毕业设计论文进行严格、具体、可执行的修改指导。

主要使用场景包括：

- 审查学生提交的整篇毕业论文 PDF。
- 按页码、章节和问题类型指出论文问题。
- 检查软件工程专业性是否充分。
- 检查需求分析、系统设计、系统实现、系统测试是否形成闭环。
- 检查图表、参考文献和明显格式问题。
- 给出可直接发给学生的导师式修改意见。
- 对论文段落进行学术表达优化。

## 总体原则

所有 Skill 必须遵守以下原则：

1. 只做毕业论文修改指导，不做选题、开题报告、任务书或项目立项。
2. 只基于用户提供的论文内容、PDF 内容或补充材料进行分析。
3. 不虚构系统功能、数据库结构、接口设计、测试数据、实验结果或参考文献。
4. 信息不足时，应明确指出缺少哪些信息，而不是替学生编造。
5. 输出应严格、具体、直接，像指导老师批改论文。
6. 不默认打分，不给百分制或等级评价，除非用户明确要求。
7. 不提供规避查重、规避 AI 检测或学术不端相关建议。
8. 可以指出口语化、模板化、重复表达和 AI 味明显的问题，但目标是提升论文质量。
9. 对 PDF 论文进行审查时，应尽量标注页码、章节和问题位置。
10. 对整篇论文进行审查时，应按章节逐项分析，不能只给总体评价。

## Skill 分工

### 1. `thesis-pdf-review-extractor`

这是 PDF 读取、结构提取和页码定位 Skill。

使用场景：

- 用户上传毕业论文 PDF。
- 用户希望按页码指出问题。
- 用户希望提取论文目录、章节、图表和参考文献。
- 用户希望定位摘要、绪论、需求分析、系统设计、系统实现、系统测试、结论、参考文献等章节。

职责：

- 读取 PDF 内容。
- 提取目录和章节结构。
- 识别主要章节所在页码。
- 识别图、表、参考文献位置。
- 为主审查 Skill 提供页码定位信息。
- 输出“页码 / 章节 / 内容摘要 / 后续审查提示”。

限制：

- 不负责最终论文质量判断。
- 不直接替代导师式审查。
- 不在无法读取清楚时强行判断。
- 对扫描版 PDF 或文字无法提取的内容，应说明限制。
- 不得在未读取到页面内容时伪造页码、章节位置或图表内容。

### 2. `software-engineering-thesis-revision-advisor`

这是主 Skill。

使用场景：

- 用户提交整篇毕业论文，希望获得导师式批改意见。
- 用户提交论文某一章，希望获得详细修改建议。
- 用户希望判断论文结构是否合理。
- 用户希望检查需求分析、系统设计、系统实现、系统测试是否符合软件工程毕业论文要求。
- 用户希望得到可直接发给学生的修改意见。

职责：

- 识别论文章节。
- 审查全文结构。
- 分章节诊断问题。
- 检查软件工程专业性。
- 检查明显格式问题。
- 给出修改建议。
- 给出下一步修改任务。
- 在信息充足时提供修改示例。

不得执行：

- 不做选题指导。
- 不做开题报告。
- 不写任务书。
- 不从零设计系统。
- 不虚构功能、数据、代码、接口、数据库和测试结果。

### 3. `thesis-reference-and-figure-checker`

这是图表与参考文献专项审查 Skill。

使用场景：

- 用户要求检查论文图表。
- 用户要求检查 E-R 图、用例图、流程图、架构图。
- 用户要求检查测试用例表。
- 用户要求检查参考文献。
- 用户希望判断正文引用和文末参考文献是否对应。

职责：

- 检查图表编号是否连续。
- 检查图题、表题是否完整。
- 检查图表是否在正文中被引用。
- 检查图表内容是否与正文描述一致。
- 检查软件工程图表是否符合论文要求。
- 检查参考文献数量、年份、格式、相关性。
- 检查正文引用和文末列表是否对应。
- 提醒需要联网核验 DOI、来源真实性的文献。

限制：

- 不伪造参考文献。
- 不补写不存在的引用。
- 不在无法联网核验时声称文献真实。
- 不替代全文综合审查。

### 4. `academic-style-rewriter-zh`

这是中文论文表达优化 Skill。

使用场景：

- 用户要求润色论文段落。
- 用户要求降低口语化表达。
- 用户要求指出 AI 味、模板化表达。
- 用户要求将内容改成正式论文语言。
- 用户要求优化摘要、绪论、结论等文字表达。

职责：

- 优化中文学术表达。
- 降低口语化、宣传化表达。
- 标记疑似 AI 化、模板化段落。
- 减少重复句式。
- 提升段落衔接和表述准确性。

限制：

- 不编造原文没有的信息。
- 不扩写不存在的功能。
- 不虚构技术方案。
- 不虚构测试数据。
- 不虚构参考文献。
- 不提供规避查重或规避 AI 检测的方法。

## 标准处理流程

当用户上传整篇 PDF 毕业论文时，按以下顺序处理：

1. 使用 `thesis-pdf-review-extractor`。
   - 提取论文目录、章节结构、页码、图表位置、参考文献位置。
   - 输出章节页码索引和可审查内容范围。

2. 使用 `software-engineering-thesis-revision-advisor`。
   - 根据提取出的章节内容进行全文导师式批改。
   - 按摘要、绪论、需求分析、系统设计、系统实现、系统测试、结论、参考文献、格式语言逐项审查。

3. 如论文包含较多图表、流程图、E-R 图、用例图、测试用例表或参考文献问题，使用 `thesis-reference-and-figure-checker`。
   - 专项检查图表和参考文献。

4. 如用户要求润色、改写或降低模板化表达，使用 `academic-style-rewriter-zh`。
   - 只对具体段落或章节进行语言优化。

当用户只提交文字章节时，可以跳过 PDF 提取步骤，直接使用 `software-engineering-thesis-revision-advisor`。当用户只要求语言改写时，使用 `academic-style-rewriter-zh`。当用户只要求参考文献或图表检查时，使用 `thesis-reference-and-figure-checker`。

## 长论文处理策略

如果论文篇幅较长，优先完成以下工作：

1. 先提取目录和章节页码。
2. 先判断全文结构是否完整。
3. 再重点审查需求分析、系统设计、系统实现、系统测试四章。
4. 最后检查摘要、绪论、结论、参考文献和语言问题。
5. 如果无法一次性完成全文细审，应明确说明已审查范围和未审查范围。

- 严格。

## 信息不足时的处理方式

如果用户提供的论文内容不足，应输出：

```markdown
当前材料不足，暂不能直接给出完整修改意见。

需要补充以下内容：

1. 论文全文或对应章节原文。
2. 系统技术栈。
3. 系统主要用户角色。
4. 系统核心功能。
5. 数据库表结构。
6. 系统测试用例。
7. 参考文献列表。
8. 图表内容或 PDF 页码截图。

在补充以上材料前，不能直接改写系统设计、系统实现和系统测试部分。
```

## 禁止事项

禁止以下行为：

- 替学生编造系统功能。
- 替学生编造数据库表。
- 替学生编造接口。
- 替学生编造测试用例结果。
- 替学生编造参考文献。
- 声称未核验的文献真实存在。
- 提供查重规避技巧。
- 提供 AI 检测规避技巧。
- 将论文审查变成单纯润色。
- 对整篇论文只给笼统评价。
- 无页码依据时伪造页码。
